# Design notes

Rationale for the load-bearing decisions in this plugin. If you're
considering a change that contradicts one of these, please open an
issue first — these were each settled after operator testing on a
live store, and the reasons aren't always obvious from the code.

## Why two grin-wallet processes (`owner_api` + `listen`)

This plugin's operator runs two long-lived `grin-wallet` processes
against the same wallet data directory:

| Process | Bind | Used for |
|---|---|---|
| `grin-wallet owner_api` | `127.0.0.1:3420` | Plugin RPC: `IssueInvoiceTx`, `CreateSlatepackMessage`, `RetrieveTxs`, `GetSlatepackAddress`, `FinalizeTx`, `PostTx` |
| `grin-wallet listen` | `127.0.0.1:3415` + Tor hidden service | Receives invoice-response slatepacks from payer wallets over Tor (e.g. `grin-wallet pay`) |

Both share the wallet's LMDB database safely — the wallet is
multi-reader internally. The two roles are intentionally split:

- **owner_api** never starts a Tor listener (BTCPay's container
  needs to reach the RPC from `172.18.0.1`, not a `.onion`).
- **listen** is the only process that registers the wallet's
  seed-derived v3 onion as a hidden service. The slatepack address
  the plugin issues to customers (`grin1...`) bech32-decodes to that
  onion's pubkey — Tor will only route to the hidden service if a
  process is currently registering it.

Operators MUST run both. Without `listen`, the plugin still
functions for the paste-back flow (customer copies response slatepack,
plugin's hosted checkout view finalizes), but `grin-wallet pay` from
the customer side will hit a dead onion and fall back to whatever
manual flow their wallet exposes. See [`SETUP.md`](SETUP.md) for the
systemd units.

## Why we run our own Tor instead of sharing BTCPay's

BTCPay's Docker stack runs its own Tor container managed by a
`docker-gen` template that auto-rewrites `torrc` on every container
event. Sharing that Tor with `grin-wallet listen` would require:

1. Pointing `socks_proxy_addr` in `grin-wallet.toml` at BTCPay's
   Tor SOCKS port
2. Adding a `HiddenService` block to BTCPay's torrc template for
   port 3415
3. **Critically:** wiring BTCPay's Tor to use grin-wallet's
   seed-derived onion key as the hidden-service identity — because
   the slatepack address bech32-encodes that specific ed25519 pubkey,
   not whatever Tor would generate on its own. Operators would have
   to copy `hs_ed25519_secret_key` files between two Tor data dirs
   and keep them in sync.

For ~20 MB RAM of duplication, grin-wallet's embedded Tor is the
upstream-supported path with one fewer integration seam to break on
BTCPay updates. Operators sufficiently motivated to consolidate are
welcome to file an issue — but the default deployment runs two Tors
deliberately.

## Why the plugin issues invoices via `IssueInvoiceTx`, not `init_send_tx`

The plugin uses receiver-initiated (invoice-flow) transactions
exclusively. Customer's wallet receives our pre-built slate, adds
inputs + partial signature, returns it; our wallet finalizes and
broadcasts.

The alternative — sender-initiated `init_send_tx` flow where the
customer's wallet builds the initial slate — would not work for
e-commerce because the resulting transaction's `tx_slate_id` is
generated by the customer's wallet. The plugin's
`GrinPaymentMonitorService` watches our own wallet for confirmations
on slates *we* issued (keyed by `tx_slate_id`); a customer-initiated
slate has an id we've never seen, so payment confirmations would
never associate back to the originating invoice.

Recipient-initiated sends (e.g. via a slatepack address QR + manual
amount entry in the customer's wallet) reach our wallet correctly
but suffer the same association problem. We currently expose only
the slatepack QR variant in the storefront UI to avoid this
footgun. Re-exposing address-based payment is gated on a monitor
enhancement that matches unassociated incoming TXs to pending
invoices by `(amount, time-window)` — tracked in [`TODO.md`](TODO.md).

## Why the storefront has its own slatepack-paste UI

When a payment integration renders its own checkout page (Medusa,
custom storefronts), bouncing the customer to BTCPay's hosted
`/stores/{id}/plugins/grin/checkout/{id}` view mid-flow for the
paste step is bad UX — different domain, different brand, breaks
the storefront's session.

The plugin exposes `POST /stores/{id}/plugins/grin/invoices/{id}/submit`
as a Greenfield-authenticated JSON endpoint accepting
`{ "responseSlatepack": "BEGINSLATEPACK..." }`. Integrators forward
the customer's pasted slatepack through that, and the plugin runs
the same decode → finalize → broadcast pipeline as the form-based
endpoint behind the hosted view.

For wallets that auto-send invoice responses over Tor (today: the
reference `grin-wallet` CLI via `grin-wallet pay`), the textarea is
unused — `grin-wallet listen` accepts the response directly. The
storefront page polls invoice status and flips to "Broadcast" on
either path.
